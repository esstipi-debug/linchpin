# рџ”— Kern

### The agentic brain for supply-chain decisions вЂ” grounded in the field's best models and sources.

**Kern** (German for *core*) is the evolution of **Linchpin** вЂ” from a tool that analyzes to the core the agency's service runs on. The name changed because the role changed: not reports on demand, but the decision kernel every engagement executes through вЂ” the QA gate that vetoes bad deliverables, citations to 25 curated sources on every result, staged writeback with rollback, and guided outcomes that always end in a safe human step. [Why Kern в†’](documentation/KERN_IDENTIDAD_Y_FILOSOFIA.md)

**Kern** turns a plain-language brief into finished, QA-gated supply-chain deliverables. A Python **engine** implements the field's established models across **38 agent-routable capabilities** вЂ” EOQ, safety stock, `(s,Q)`/`(R,S)` policies, multi-echelon, simulation, forecasting, pricing, price intelligence (competitor position), DDMRP, ABC-XYZ, sourcing, landed cost, cost-to-serve, S&OP, facility location, DRP, transportation, FEFO, reconciliation, slotting, warehouse layout, vehicle routing and more вЂ” and an **orchestrator agent** drives every one of them end to end with a **never-unprotected guarantee** (every result is executed *or* hands you a ready, safe next step) and **safe-staging writeback**. Each result is **grounded** in a knowledge graph of **25 curated SCM sources and the codebase itself**.

## вљЎ What it does

Give it a brief; it **classifies в†’ runs в†’ validates (QA) в†’ delivers**. If QA fails, nothing ships.

```mermaid
flowchart LR
  A["Brief (+ optional data)"] --> C{"Classify intent"}
  C --> I["inventory_optimization"]
  C --> P["pricing"]
  C --> L["leadership_chain"]
  C --> M["...31 more tools"]
  I --> Q{"QA gate"}
  P --> Q
  L --> Q
  M --> Q
  Q -->|pass| G["Ground in L3 вЂ” source + code citations"]
  G --> D["Deliverables вЂ” Excel В· report В· chart + Fuentes"]
  Q -->|fail| N["no deliverable"]
```

Every one of the **38 registered tools** below is routed the same way вЂ” one `register()` call wires a capability into the same `classify в†’ prepare в†’ run в†’ QA в†’ deliver` pipeline, no special-casing. A few, as a sample:

| Capability | Input | Deliverable |
|---|---|---|
| рџ“¦ `inventory_optimization` | demand CSV/Excel | Excel + report + CSV вЂ” forecast в†’ `(s,Q)`/`(R,S)` в†’ budget fit |
| рџ’І `pricing` | price/qty CSV/Excel | Excel + report вЂ” elasticity в†’ margin-maximizing price |
| рџ§­ `leadership_chain` | a brief / scores | radar chart + report вЂ” CHAIN leadership profile + directives |
| рџЏ—пёЏ `warehouse_layout` | params / brief | 3D HTML + layout.json + report вЂ” navigable warehouse (building, yard, docks, gates, racks) |

Runs **with or without an LLM**: an optional `LLMProvider` (Claude) sharpens routing and the narrative; the deterministic core works on its own. The whole thing is **1100+ tests**.

Three cross-cutting guarantees keep the agent safe in production:

- **Never-unprotected** вЂ” every result is `EXECUTED` or carries an executable path: ranked options, a prepared handoff (pre-filled PO / email / count sheet), or an escalation. No dead ends.
- **Safe-staging writeback** вЂ” changes are computed as a dry-run changeset, gated by risk tier + time-boxed approval, applied idempotently, and fully auditable / `rollback()`-able. The agent never mutates a system of record blindly.
- **Data protection by default** вЂ” every upload is validated (size, type, path) and processed in an isolated, auto-purged per-job directory; no secrets are ever committed. Full threat model and controls in [SECURITY.md](SECURITY.md).

## рџ§  The agent (`scm_agent/`)

A **registry-based orchestrator**: every capability is a `Tool` with four stages вЂ” `prepare в†’ run в†’ qa в†’ deliver` вЂ” that the orchestrator drives, enforcing **"QA fails в‡’ no deliverable"** in one place. Adding a capability is one `register()` call; no routing edits.

```
brief в”Ђв–¶ intent.classify в”Ђв–¶ registry.get(tool) в”Ђв–¶ prepare в”Ђв–¶ run в”Ђв–¶ QA в”Ђв–¶ deliver в”Ђв–¶ JobResult
              (rules + optional LLM)              any of the 38 registered tools
```

- **`scm_agent/`** вЂ” `types` В· `llm` (Claude / rules fallback) В· `registry` В· `tools` В· `intent` В· `orchestrator` В· `knowledge` (L3 grounding) В· `guided_bridge` (never-unprotected contract via `src/guided.py`)
- **Entry points** вЂ” CLI `examples/run_agent.py`, HTTP `POST /api/jobs` (multipart, with downloadable deliverables), and the live console under `webapp/static/prototype/`
- **Statuses** вЂ” `ok` В· `needs_clarification` В· `needs_data` В· `qa_failed` В· `error`

Full reference: [`scm_agent/README.md`](scm_agent/README.md). The `leadership_chain` capability scores a brief against the **CHAIN** model (Collaborative / Holistic / Adaptable / Influential / Narrative) вЂ” an original synthesis, not a reproduction of any single external text.

## рџ§  L3 вЂ” domain knowledge & the theoryв†”code bridge

Every job is **grounded**: the orchestrator queries a knowledge graph and attaches citations to each result (the **Fuentes** shown in the console). Two graphs, one read-only query surface вЂ” [`scm_agent/knowledge.py`](scm_agent/knowledge.py):

- **Domain knowledge graph** ([`knowledge/scm-books/`](knowledge/scm-books/README.md)) вЂ” **25 curated SCM sources** (forecasting, pricing, revenue management, inventory, operations, logistics, sustainability, leadership, AI in supply chains): ~1,950 concept nodes with source citations. Committed.
- **Code graph** (`graphify-out/`) вЂ” the codebase itself, built with `/graphify`. Gitignored (regenerable).

The **bridge** ties them together: for each cited concept it resolves the `src/` module that implements it, so a deliverable cites the concept **and** the function behind it.

```text
Economic Order Quantity           ->  src/eoq.py
Safety Stock                      ->  src/safety_stock.py
Cost & Service-Level Optimization ->  src/cost_optimization.py
```

Query it directly: `python examples/query_knowledge.py --bridge "newsvendor"` В· `--search "fill rate"` В· `--explain crostons_method`.

## вљ™пёЏ The engine вЂ” one question, one lens

The analytical core the agent stands on: a chain of small **pure functions**, each answering one question with one point of view. Every number is simulation-real вЂ” no Node, no build step.

<div align="center">

![The engine pipeline: PREDICT (forecasting, distributions) в†’ DECIDE (eoq, policies, safety_stock) в†’ OPTIMIZE (fill_rate, cost_optimization, simulation_opt) в†’ REALITY (multi_echelon, newsvendor, constraints) в†’ a QA-gated plan, every number cross-checked against Monte-Carlo simulation](docs/assets/engine-lenses.svg)

</div>

| The question | Engine | The lens it brings |
|---|---|---|
| *How much do I order?* | `eoq.py` | **Cost balance** вЂ” the lot size where ordering cost meets holding cost (+ volume discounts). |
| *When do I reorder?* | `policies.py` | **Trigger & risk** вЂ” forecast + costs в†’ `(s,Q)`/`(R,S)`: the reorder point *and* the quantity. |
| *How big a buffer?* | `safety_stock.py` | **Uncertainty** вЂ” `zВ·ПѓВ·в€љП„`, keyed off **forecast error Пѓ_e**, not raw demand spread (the classic mistake). |
| *What service do I actually get?* | `fill_rate.py` | **Realized service** вЂ” fill rate ОІ (units served), which stays high even when cycle service О± drops. |
| *Is more service worth it?* | `cost_optimization.py` | **Economics** вЂ” the service level / review period that minimizes holding + shortage cost, not a rule of thumb. |
| *Spiky or skewed demand?* | `forecasting.py` В· `distributions.py` | **Shape-aware** вЂ” Croston for intermittent, gamma for skew, exactly where the normal model under-stocks. |
| *Where to hold stock across the network?* | `multi_echelon.py` | **System view** вЂ” guaranteed-service placement across stages, not greedy per node. |
| *One perishable shot?* | `newsvendor.py` | **Single period** вЂ” the critical ratio `cu/(cu+co)`. |
| *Can't trust the assumptions?* | `simulation_opt.py` | **Empirical** вЂ” simulate thousands of demand paths, search `(s,Q,S)` for the real optimum. |
| *Is the plan feasible?* | `constraints.py` | **Operational reality** вЂ” MOQ, case packs, shelf-life, and a budget allocator that trims safety stock to fit. |
| *What price maximizes margin?* | `pricing.py` | **Elasticity** вЂ” per-SKU price sensitivity в†’ the margin-maximizing markup. |

**Why this beats a spreadsheet**

![simulation validated](https://img.shields.io/badge/policies-simulation--validated-4fd1c5?style=flat-square&labelColor=0d1219)
![forecast error](https://img.shields.io/badge/safety_stock-forecast_error_Пѓ-5eead4?style=flat-square&labelColor=0d1219)
![shape aware](https://img.shields.io/badge/demand-Croston_and_gamma-3fb950?style=flat-square&labelColor=0d1219)
![feasible](https://img.shields.io/badge/plan-MOQ_and_budget_feasible-d4a017?style=flat-square&labelColor=0d1219)
![tests](https://img.shields.io/badge/1100%2B_tests-passing-4fd1c5?style=flat-square&labelColor=0d1219)

- **Validated, not assumed** вЂ” closed-form policies are cross-checked against Monte-Carlo simulation (`simulation.py`, backorders + lost sales).
- **Пѓ_e, not Пѓ_demand** вЂ” safety stock keys off *forecast error*, the only correct dispersion (the #1 inventory mistake in the wild).
- **Shape-aware** вЂ” intermittent (Croston) and skewed (gamma) demand are first-class, where a normal-curve sheet silently stocks out.
- **Feasible by construction** вЂ” MOQ / budget are constraints, so what ships is buildable, not just mathematically optimal.
- **Pure & composable** вЂ” each module is a pure function, the engine validated against reference numeric examples in a 1100+ test suite; the orchestrator chains them without surprises.

## рџЏ—пёЏ From engine to product

The full chain runs end to end (`examples/run_constrained_plan.py`):

```
data source в†’ forecast (Пѓ_e) в†’ (s,Q)/(R,S) policy в†’ MOQ/case packs в†’ budget fit
src/sources.py   src/forecasting.py   src/policies.py   src/constraints.py
```

- **Pluggable data** (`src/sources.py`) вЂ” CSV, in-memory DataFrame, or any SQL database via `SqlDemandSource` (SQLite, Postgres, MySQL). New backends just satisfy the `DemandSource` protocol.
- **Forecasting** (`src/forecasting.py`) вЂ” MA / SES / Croston, exposing Пѓ_e, the correct safety-stock dispersion.
- **Constraints** (`src/constraints.py`) вЂ” MOQ, case packs, shelf-life caps, and a budget allocator that trims safety stock across the portfolio to fit.
- **Web UI** (`webapp/`) вЂ” a 4-tab planner (Portfolio В· SKU Detail В· Budget Planner В· Forecast Quality) + the live agent console, served by FastAPI over the engine. See [webapp/README.md](webapp/README.md).
- **Job-fulfillment layer** (`jobs/`) вЂ” turn a client's demand file (any schema) into client-ready Excel + a written report with automated QA. See [jobs/README.md](jobs/README.md).
